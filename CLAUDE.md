# SmartFlow — крипто-аналітика: Telegram бот + Mini App

## Контекст проєкту
Повна специфікація: docs/SPEC.md — ЗАВЖДИ звіряйся з нею перед реалізацією.
Плейбук задач по етапах: docs/PROMPTS.md.
Журнал прийнятих рішень: docs/DECISIONS.md — додавай туди запис після кожного
важливого архітектурного рішення.

## Стек
- Backend: Python 3.11+, FastAPI, SQLAlchemy 2.x (async), Alembic, aiogram 3,
  ccxt, pandas, pandas-ta, smartmoneyconcepts, vectorbt, APScheduler,
  pydantic-settings, slowapi
- БД: SQLite (старт) → PostgreSQL (без зміни коду, тільки connection string)
- Frontend (Mini App): vanilla JS + Lightweight Charts, деплой Cloudflare Workers
- Прод: Ubuntu 22.04 ARM (Oracle Cloud), systemd-сервіси
- AI: Google Gemini API (сентимент новин і пояснення сигналів)

## Середовище розробки
- Windows + PowerShell. НЕ використовуй head/tail/grep у командах для
  локальної машини — використовуй Get-Content / Select-String.
  Кирилиця: завжди UTF-8.
- Python строго 3.11-3.12 (НЕ 3.13+): pycares/aiodns і numba несумісні з
  новішими версіями; venv створювати через py -3.12.
- Деплой на сервер: scp. ВАЖЛИВО: після scp завжди перевіряй, що файл
  реально оновився (через хеш або grep -c характерного рядка на сервері) —
  scp інколи мовчки лишає старий файл.

## Правила коду
- Повні файли без заглушок, без "..." і без TODO-пропусків
- Type hints всюди; pydantic-моделі для всіх вхідних/вихідних даних API
- Кожен модуль супроводжується pytest-тестами; ЗАПУСКАЙ тести перед тим,
  як вважати задачу завершеною
- Зовнішні API (біржі, новини, Gemini, Telegram): retry з exponential
  backoff, таймаути, ніколи не дозволяй винятку покласти сервіс
- Логування: structlog або logging з JSON-форматом; У ЛОГИ НІКОЛИ не
  потрапляють API-ключі, секрети, токени (фільтр обовʼязковий)

## Правила безпеки (інваріанти, порушувати заборонено)
- Секрети тільки з .env / /etc/smartflow/secrets.env — ніколи в коді чи git
- API-ключі бірж у БД тільки зашифровані AES-256-GCM
- Ключі з правом withdrawal — відхиляти при додаванні
- Стоп-лоси і тейк-профіти — ТІЛЬКИ біржові ордери (нативні TP/SL),
  ніколи не програмні (що залежать від аптайму нашого сервера)
- Тільки ORM / параметризовані запити, жодного сирого SQL з конкатенацією
- Кожен ордер має clientOrderId = UUID з нашої БД (ідемпотентність)
- Валідація Telegram initData (HMAC-SHA256) на кожен запит API

## Фінансова логіка — особливий режим
Файли backend/app/trading/risk.py та executor.py:
- пиши максимально просто і прямолінійно, без "розумних" скорочень
- тести на крайні випадки обовʼязкові (нульовий депозит, стоп == вхід,
  відʼємні значення, екстремальне плече)
- познач у коментарі: "REVIEW: перевіряється людиною"

## Команди
- Тести: cd backend && pytest -q
- Локальний запуск API: cd backend && uvicorn app.main:app --reload
- Міграції: alembic revision --autogenerate -m "..." && alembic upgrade head

## Git і якість (проєкт — флагман портфоліо, це впливає на вимоги)
- Conventional commits АНГЛІЙСЬКОЮ: feat:, fix:, test:, docs:, refactor:,
  ci:. Маленькі коміти після кожної робочої фічі, осмислені повідомлення
- Docstrings і коментарі в коді — англійською (код міжнародний);
  спілкування з користувачем у чаті — українською
- CI (GitHub Actions) має бути зеленим: pytest + ruff + mypy. Якщо твоя
  зміна ламає лінтер чи типи — виправ перед завершенням задачі
- Покриття тестами: ціль 85%+ загалом; risk.py та security/crypto.py — 100%,
  включно з крайніми випадками
- Не коміть: .env, *.db, __pycache__, node_modules
- Після завершення кожної задачі і успішного проходження тестів —
  git add, коміт за conventional commits і git push origin main.
  Якщо push не вдався — повідом про це, не ігноруй.
